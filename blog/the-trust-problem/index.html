Trust & accuracy

Smarter Models Make Data Generation Worse, Not Better

Why “Almost Right” Isn’t Good Enough for Business Data

Ruth Guan July 2, 2026 4 min read

Data work has never really been about generating the SQL. It’s about generating the right SQL to get the correct answer. That takes two things: knowing the SQL code, and knowing the business semantics. AI has taken care of the SQL code part, but the business context — knowing which tables, columns, and definitions to use — is still imperfect. And getting the data “almost right,” or right 99% of the time, still isn’t good enough.

What’s dangerous is the answer that’s plausible, clean, well-formatted, and wrong.

  • You ask for “revenue,” and the query quietly folds in tax and refunds. The number still looks like revenue.
  • You ask about “last quarter,” and it resolves to the wrong date range, using calendar year instead of fiscal year. The trend line still draws beautifully.
  • You ask for customers by department, and two tables get joined on a key the model guessed at, linking the wrong records. The totals land in a believable range, so no one blinks.

None of these throw an error. Every one of them produces a confident, professional-looking answer headed straight for a slide. A wrong answer that looks wrong costs you nothing — you catch it. A wrong answer that looks right costs you a decision.

It’s a meaning problem, not a language problem

A language model knows language. It does not know your business. It doesn’t know that we count an “active customer” as someone who bought in the last 90 days, not anyone who ever signed up. It doesn’t know that when we say “clients,” the database calls them “accounts.” Out of the box, it fills every one of those gaps with a confident guess.

And the better it writes, the more convincing those guesses get. Improving the model alone doesn’t earn trust. It just raises the stakes of getting the meaning wrong. This was never a language problem. It’s a meaning problem.

So what actually earns trust?

If AI solved the SQL but not the meaning, then earning trust comes down to three things.

  1. Consistency. Ask the same question three different ways and you should get the same answer. If you get three different numbers, you don’t have an accuracy solution — you have a fluency problem. That means the business’s meaning has to live in one shared place, so “revenue” means one specific thing everywhere: every chat, every report, every dashboard. When meaning is shared, answers stop drifting.
  2. Validation. I don’t want a system that hands me every answer with the same unearned confidence. I want one that checks its own work, and when something looks off, says so and asks instead of quietly serving me a wrong number. I’ll take a system that occasionally pauses to confirm over one that’s smoothly, fluently wrong.
  3. Explainability. Trust isn’t “believe me.” It’s “here’s how I got here — check it.” Show me the query that actually ran, the tables and filters it used, and a plain-English read of what the numbers mean. If I’m not technical, I get an explanation I can follow. If I am, I get the SQL to verify. Either way the answer is something I can inspect, not a black box I have to have faith in.
